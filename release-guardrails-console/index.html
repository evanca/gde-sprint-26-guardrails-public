This codelab is part of the 2026 Agentic Architect Sprint by Google.

An architect will not point an autonomous agent at a release repository on trust. Before it runs, you want a guarantee that it cannot do something catastrophic — read a production secret, rewrite an applied migration, force-push a shared branch, install an unvetted package, or delete a live deployment. This codelab builds that guarantee with the Antigravity SDK (google-antigravity, Python): a deny-by-default policy, a disabled raw shell, a curated tool that makes a risky command safe, and an audit trail of everything that ran.

The starting point

The Release Guardrails Console is a mock release repo — services by environment, feature flags, applied migrations, production config, and masked secrets, all in local JSON. No real secrets, backend, or network. An autonomous agent works inside it through the SDK.

In the starter, the guard logic (guard/policy.py) is an unimplemented stub, so the agent is effectively ungoverned: it can read the sealed secret and issue a real kubectl delete.

The problem

Agentic SDKs let an agent call tools — read and edit files, run commands, talk to a cluster. That power is the risk. An ungoverned agent asked to "free up space" might delete a migration; asked to "verify the database" might print a secret. You need to intercept every tool call before it executes and decide what is allowed — without hand-writing a bespoke check for each one.

How the Antigravity SDK helps

The SDK gives you a declarative policy system and a lifecycle-hook pipeline that runs on every tool call: Decide → Execute → Inspect.

  • Decide (deny-by-default): a policy list — deny_all() plus explicit allow(...), with when= predicates on the arguments — blocks dangerous calls and returns a reason the agent can read and adapt to.
  • Disable: CapabilitiesConfig removes a whole tool (raw shell) from the agent's reach, so some commands are not merely denied — they are impossible.
  • Rewrite before execution: raw shell is gone, so the only way to the cluster is a curated kubectl tool whose own code rewrites a destructive delete into --dry-run=client before it runs.
  • Inspect (audit): a post_tool_call hook records everything that executed.

What You Will Build

A release repo an autonomous agent can work in safely:

  • a deny-by-default policy that denies reading sealed secrets and editing protected paths (migrations, secrets), each with a reason;
  • raw run_command disabled entirely, so force-push and package installs are unavailable;
  • a curated kubectl tool that rewrites delete to --dry-run=client;
  • a post-tool audit trail of every tool the agent ran.

What You Will Learn

  • How the SDK's hook lifecycle (Decide → Execute → Inspect) intercepts tool calls.
  • How to express deny-by-default with policy.deny_all() + allow(...) + when= predicates.
  • When to disable a tool (capabilities) versus deny it (policy), and why.
  • How a denial returns a reason the agent adapts to, instead of failing silently.
  • How a curated tool rewrites a risky command before execution.
  • How to unit-test a safety policy deterministically, offline.

Cost and Scope

The guard logic (guard/policy.py) is pure Python, so the whole policy is tested offline with python -m unittest — no agent, no network, no SDK install needed to verify it. Running the live agent against the guard (with the SDK installed) is bounded: a couple of short prompts. There is no real cluster — the kubectl tool reports what it would run.

guard/agent.py turns that pure logic into a live guard using the SDK's real policy and hook API. Read it — three mechanisms work together:

from google.antigravity import Agent, CapabilitiesConfig, LocalAgentConfig, types
from google.antigravity.hooks import hooks, policy
from .policy import is_protected, is_secret
from .tools import kubectl

@hooks.post_tool_call                      # Inspect: audit everything that ran
async def release_audit(data) -> None:
    print(audit_line(data))

policies = [
    policy.deny_all(),                                            # deny by default
    policy.allow(types.BuiltinTools.VIEW_FILE.value),            # reads...
    policy.deny(types.BuiltinTools.VIEW_FILE.value,              # ...except secrets
               when=lambda a: is_secret(_path(a)), name="no-read-secrets"),
    policy.allow(types.BuiltinTools.EDIT_FILE.value),           # edits...
    policy.deny(types.BuiltinTools.EDIT_FILE.value,             # ...except protected paths
               when=lambda a: is_protected(_path(a)), name="protect-edit"),
    policy.allow(kubectl.__name__),                              # the curated cluster tool
]

config = LocalAgentConfig(
    system_instructions="You are a release engineer. Raw shell is unavailable; "
                        "use the kubectl tool. If an action is denied, adapt.",
    capabilities=CapabilitiesConfig(disabled_tools=[types.BuiltinTools.RUN_COMMAND.value]),
    tools=[kubectl],
    policies=policies,
    hooks=[release_audit],
)

Disable vs. deny. CapabilitiesConfig.disabled_tools strips a tool from the model's context entirely (it never sees run_command), while policy.deny(...) leaves the tool visible but blocks the call at runtime with a reason the agent can read. Disable what the agent should never use; deny what depends on the arguments.

The kubectl tool (guard/tools.py) is the only path to the cluster, and it sanitizes its own arguments — a delete becomes --dry-run=client before it runs. That is the one place a command is genuinely rewritten before execution.

Run the agent again with the guard implemented:

python -m guard.agent

Observe the same three-part request, now governed:

  • The secret read is denied with a policy reason the agent reports back.
  • The flag edit is allowed — deny-by-default is not deny-everything.
  • The kubectl delete is rewritten to append --dry-run=client; nothing is actually deleted.
  • Try to make it force-push or npm install: it cannot — raw shell is disabled, so those tools are not even available.

The post-tool audit (Inspect) hook leaves a line per tool the agent actually ran — the reviewable record an architect needs.

You put runtime guardrails on an autonomous agent with the Antigravity SDK: a deny-by-default policy that blocks reading secrets and editing migrations with a reason, a disabled raw shell that makes force-push and installs impossible, a curated tool that rewrites a destructive kubectl delete into a dry run before it runs, and an audit trail of everything that executed — with the decision logic verified offline.

You learned:

  • Deny-by-default gives the agent an explicit allowlist, not blanket trust.
  • The hook lifecycle is Decide → Execute → Inspect; a Decide policy returns a reason the agent adapts to.
  • Disabling a tool (capabilities) and denying a tool (policy) solve different problems.
  • A curated tool is where a risky command gets rewritten safely before execution.
  • Safety policies are plain programs, so they are testable like any other code.

Policies are lifecycle hooks underneath. The declarative policy module is a shortcut: policy.enforce(...) compiles your allow/deny/ask rules into a PreToolCallDecideHook — the same pre-tool Decide hook you could write by hand. For logic the builders don't express, implement hooks directly with the decorators (@hooks.pre_tool_call_decide to allow/deny, @hooks.post_tool_call to inspect, @hooks.on_tool_error to recover); the audit in this lab is one such hand-written hook. Reach for policies for declarative guardrails, and raw hooks when you need custom behavior.
